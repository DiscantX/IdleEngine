# Progress

## Current status

- Completed the core engine slices through minimal save/load, a second system type (decay), the Formula/Value math primitives, and now a full Definitions/Upgrades/ModifierSystem layer supporting live-resolved, purchasable upgrades that modify production and decay rates.
- Verified deterministic simulation behavior, offline advancement, full save/load round-tripping, multi-system interaction, Value/Formula resolution, and now end-to-end upgrade purchasing with correctly-folded modifier effects.

## Recently completed

- Implemented the engine foundation: GameState, Entity, ProductionComponent, EntityAPI, ComponentAPI, ResourceAPI, and ProductionSystem.
- Added a pluggable TimeSource abstraction and LocalTimeSource.
- Added offline progression support through Clock.advanceOffline() and Engine.loadOffline().
- Established a folder convention where interfaces live under their own interfaces/ subdirectory.
- BigNumber integration is complete: GameState.resources, ProductionComponent.rate, ResourceAPI, and ProductionSystem all operate on BigNumber.
- Minimal save/load is complete: SaveData (versioned GameState wrapper), Serializer (GameState/SaveData <-> JSON, with generic BigNumber tagging via a replacer/reviver so it works anywhere in the state tree, not just resources), SaveEncoder (JSON <-> checksummed blob, base64 + djb2 checksum, throws on corruption/tamper detection), StorageAdapter interface + LocalStorageAdapter (pluggable persistence, same pattern as TimeSource), and SaveManager (orchestrates the pipeline, with saveKey/saveVersion passed in by the game rather than hardcoded by the engine). Engine now owns a SaveManager and exposes save()/load() facade methods.
- Verified end to end in main.ts: ticking a game, saving it, loading into a separate fresh instance, and confirming BigNumber values survive the round trip intact.
- Slice 7: added DecaySystem as a second system type, to stress-test multi-system architecture before moving into the bigger Formula/Value/Definitions design work. DecayComponent (resource, rate, optional clampAtZero defaulting to true) mirrors ProductionComponent's shape. DecaySystem subtracts rate * deltaTime from a resource each tick, clamping the actual subtraction to never exceed the current resource amount when clampAtZero is true (via new BigNumber.min()/max() helpers), using ResourceAPI.add() with a negated amount rather than adding a new subtract method to ResourceAPI. Registered alongside ProductionSystem in GameSetup.ts's Simulation, with a new decay-only goldVault entity added to isolate decay behavior from production. Verified end to end in main.ts: production (25/sec) and decay (5/sec) both acting on the same gold resource in one tick produced the correct net result (200 gold after 10s), confirming systems run in registration order and interact correctly on shared state.
- Slice 8: added Formula (type alias, `(level: number) => number | BigNumber`) and Value (class wrapping a discriminated union `ValueSource` of `{kind: "constant", value: BigNumber}` or `{kind: "formula", formula: Formula}`, with private constructor, static factories `Value.constant()`/`Value.fromFormula()`, and a single `resolve(level = 0): BigNumber` method). Chose function-based Formulas over data-driven ones: Formulas only ever live on Definitions (static content, never part of GameState, never touched by Serializer), so the serializability argument for a data-driven interpreter didn't apply, and predicting every formula shape up front isn't feasible. Verified in main.ts: a constant Value resolving identically regardless of level argument, and a formula-backed Value (`10 * 1.15^level`, an upgrade-cost-style curve) resolving correctly at levels 0/5/10.
- Slice 8 follow-up: closed the deferred BigNumber ergonomics item. Added `BigNumber.from(value: number | BigNumber): BigNumber`, a static normalization helper (passes through an existing BigNumber, otherwise delegates to `fromNumber`). Relaxed `Formula`'s return type and `Value.constant()`'s parameter type to `number | BigNumber`, normalizing via `BigNumber.from()` at those two boundary points only. `ValueSource` and `Value.resolve()` remain strictly `BigNumber` internally, so the ergonomics only loosen at the exact points a game dev supplies a number — everything downstream (systems, APIs, BigNumber arithmetic) stays untouched and fully strict. Re-verified in main.ts with the same test cases rewritten using plain numbers only (no explicit BigNumber construction), confirming identical resolved output to the original explicit version.
- **Slice 9: Definitions, Upgrades, and ModifierSystem.** Built the full static-content and modifier layer on top of the Formula/Value foundation:
  - `Definition` (shared minimal contract: `id`, `name`, `description?`) added to `engine/data/Definitions.ts`, deliberately kept to universal display/identity fields only — see `design.md`'s new "Extending Definitions with Game-Specific Metadata" section for the reasoning (typed interface extension in game code, not a generic metadata bag).
  - `ModifierOperation` (`"add" | "multiplyAdditive" | "multiplyCompound"`) and `ModifierDefinition` (`target`, `operation`, `magnitude: Value`) added. `target` is an opaque, game-defined string convention (e.g. `"production:gold"`) that calling systems construct and ModifierSystem only matches — the engine has no built-in notion of what a target means.
  - Three-operation design was deliberately expanded from an initial two (`add`/`multiply`) after recognizing that "multiply" has a real combining-order ambiguity that "add" doesn't: multiple percentage bonuses on the same target can either stack additively (two +10% upgrades → +20%, the common case) or compound independently (two +10% upgrades → +21%). `multiplyAdditive` and `multiplyCompound` were split out to let games express either behavior explicitly per-modifier, rather than the engine picking one and forcing every game into it.
  - `ResourceCost` (`resource`, `amount: Value`) added after recognizing a single `cost`/`costResource` pair on `UpgradeDefinition` couldn't express multi-resource costs (e.g. 100 gold + 50 wood). `UpgradeDefinition.costs` is a `ResourceCost[]`, mirroring how `modifiers` is already a list.
  - `UpgradeDefinition` (`extends Definition`: `costs: ResourceCost[]`, `maxLevel?`, `modifiers: ModifierDefinition[]`) added.
  - `BuildingDefinition` (`extends Definition`: `components: {[componentType: string]: object}`) added, and `game/definitions/buildings.ts`'s `goldMine`/`goldVault` retyped from untyped object literals to explicit `BuildingDefinition`s with `name`/`description`, closing a consistency gap where building content didn't share the same contract as upgrade content.
  - `GameState.upgrades: {[upgradeId: string]: number}` added, following the same sparse-map convention as `resources` (absent key = level/amount zero; keys created lazily on first write, never pre-populated).
  - `UpgradeAPI` (`getLevel()`, `purchase()`) added to `engine/api/`. `purchase()` does a two-pass check-then-deduct over `costs` (resolve and verify every cost is affordable *before* deducting any of them, to avoid partial charges if a multi-resource purchase can only partially be paid), then increments the level. Constructor-injects `ResourceAPI`, following the established "systems own their dependencies" pattern.
  - `ModifierSystem` (`engine/systems/`) added — not a `SimulationSystem` (doesn't run in the `Simulation` tick list), but a query service other systems call: `getEffectiveValue(state, target, baseValue)` scans all owned upgrades' definitions for modifiers matching `target`, and folds them into `baseValue` in three deterministic stages (all `add` modifiers summed and added; all `multiplyAdditive` modifiers summed into one combined bonus multiplier; each `multiplyCompound` modifier applied as its own independent, compounding multiplier) — chosen specifically so purchase *order* never affects the result. Constructor-injects the full list of `UpgradeDefinition`s and an `UpgradeAPI` instance.
  - Modifiers are resolved live, every call, rather than mutating components at purchase time — confirmed in Slice 9's end-to-end test, where the raw `ProductionComponent.rate` in the printed GameState remained the unmodified base value (25) even after a modifier was actively boosting effective output to 27.5.
  - `ProductionSystem` and `DecaySystem` both updated to take a third constructor-injected `ModifierSystem` and resolve `getEffectiveValue(state, \`production:${resource}\`, rate)` / `getEffectiveValue(state, \`decay:${resource}\`, rate)` in place of reading `rate` raw — with DecaySystem's fold happening *before* the existing clampAtZero logic, so a modifier that reduces decay is clamped against the already-reduced amount, not the original.
  - `Engine` gained a fourth constructor-injected dependency (`UpgradeAPI`) and a `purchase(definition): boolean` facade method, mirroring how `save()`/`load()` already wrap `SaveManager` rather than exposing it directly.
  - Verified end to end in main.ts: seeded 10 gold directly (test-only shortcut, bypassing simulation for a known starting balance), purchased one level of a `miningSpeed` upgrade (+10% multiplyAdditive to `"production:gold"`, costing 10 gold), ticked 10 seconds, and confirmed the result matched hand-calculated expectations exactly (225 net gold: 27.5/sec effective production − 5/sec unmodified decay, over 10s), with the upgrade level correctly recorded and the raw component rate confirmed untouched.
  - **Slice 9 follow-up:** extended the `number | BigNumber` ergonomics pattern from Value/Formula to `ResourceAPI.add()` — `amount` now accepts either type, normalized via `BigNumber.from()` at the top of the method, so game/system code calling `resourceAPI.add()` directly no longer needs to explicitly construct a BigNumber for simple literal amounts. (Note: this doesn't affect main.ts's direct `state.resources.gold = ...` test-setup assignment, since that bypasses ResourceAPI entirely — confirmed deliberately, not a gap.)
  - **Slice 9 follow-up 2: `UpgradeAPI.canPurchase()`.** Added a check-only method for UI code to query "would `purchase()` succeed right now" (checks max-level and affordability, no mutation) without duplicating logic. Extracted a private `evaluatePurchase()` helper shared by both `canPurchase()` and `purchase()`, so the check logic exists in exactly one place, and `purchase()` resolves each cost `Value` exactly once per call (reusing `evaluatePurchase()`'s resolved amounts for deduction) rather than the double-resolve its earlier draft had.
  - **Slice 10 (engine-only): Purchasable generalization, Producer content type, EntityDefinition rename.** Triggered by planning the example game's content and discovering three real gaps: no centralized purchase logic reusable beyond upgrades, no way to represent quantity-based content that isn't an entity, and `BuildingDefinition` was a slightly-too-game-flavored engine name.
  - `Purchasable` (`costs: ResourceCost[]`, `maxCount?: number`) added to `engine/data/Definitions.ts` as a shared interface. `UpgradeDefinition` now extends it (dropping its own `costs`/`maxLevel`, gaining `maxCount` in their place).
  - `BuildingDefinition` renamed to `EntityDefinition` (engine-neutral naming — "building" implied a city-builder-genre assumption the engine shouldn't make) and now also extends `Purchasable`.
  - `ProducerDefinition` (new, extends `Definition, Purchasable`: `resource: string`, `rate: Value`) added — represents quantitative, count-based, purchasable content that generates a resource, with no individual identity (owning several is one shared count, not separate entities).
  - Landed on a deliberate identity distinction to decide entity vs. producer shape going forward: entities have **qualitative identity** (individually stored/addressable, even if property-identical to another); producers (and upgrades) have **quantitative identity** (a single definition + count, no individual instances). This is now the standing rule for choosing between the two patterns for any future content type.
  - `Entity` gained an optional `definitionId?: string` field, letting entities trace back to the `EntityDefinition` they were acquired from — enables cap-checking against `Purchasable.maxCount` via a query rather than a separate count-tracking map to keep in sync (entities remain sole source of truth for their own existence).
  - `GameState.producers: {[producerId: string]: number}` added, sparse map convention matching `upgrades`.
  - Extracted a standalone `evaluatePurchase()` function (`engine/api/Purchasing.ts`) — generalized version of what was previously a private method on `UpgradeAPI`, taking `currentCount` as a parameter so it has no opinion on what "count" means for a given content type. Deliberately a plain function, not a shared base class — a base class was considered and rejected specifically because it would've introduced inheritance, a pattern not used anywhere else in the codebase (every other shared-behavior case uses constructor-injected composition instead).
  - `UpgradeAPI` refactored to call the standalone function instead of its own private copy; behavior unchanged.
  - `ProducerAPI` (new, mirrors `UpgradeAPI`'s shape exactly): `getQuantity()`, `canPurchase()`, `purchase()`, using the same standalone `evaluatePurchase()`.
  - `ProductionSystem` now reads from two sources each tick: the existing entity-component loop (unchanged), plus a new loop over constructor-injected `ProducerDefinition[]` × `state.producers` counts (via `ProducerAPI.getQuantity()`), computing `perUnitRate * quantity` as the base rate before folding through the same `ModifierSystem` target convention (`production:${resource}`) entity-based production already used — confirming Option A (modifiers are resource-scoped, not per-producer-source) applies uniformly to both.
  - `QueryAPI` implemented for the first time: `countEntitiesByDefinition(state, definitionId)`, a plain scan over `state.entities` matching `definitionId`. Not yet called by anything (no entity-purchasing path exists yet) — foundation only.
  - `game/definitions/buildings.ts`'s `goldMine`/`goldVault` updated to the `EntityDefinition` type with `costs: []` (honest placeholder — neither is currently acquired through any purchase flow; both are still created directly in `GameSetup.ts` at startup, unchanged from Slice 9).
  - `GameSetup.ts` updated: `producers: {}` added to initial `GameState`; `ProducerAPI` constructed; `ProductionSystem` now receives `producerDefinitions` (currently an empty array — no producer game content exists yet, deliberately deferred to next slice) and `producerAPI`. `Engine` itself still only wires through `upgradeAPI`/`purchase()` for `UpgradeDefinition` — no generic purchase facade yet, since nothing calls `producerAPI.purchase()` until real producer content exists.
  - Verified end to end in `main.ts`: Slice 9's Game G test re-run unchanged, confirming entity-based production/decay behavior is untouched (225 gold after 10s) and the new producer loop is inert-but-present (empty array, contributes nothing) rather than broken.

## Deferred / follow-up

- System clock resilience is partially handled with negative-time clamping and max elapsed caps.
- External or server-based time sources remain deferred until networking and save infrastructure are in place.
- Save migrations (Migration.ts) are deferred; SaveData.version exists and is stamped on every save, giving future migration logic something to key off, but no migration logic has been written yet.
- SaveEncoder's checksum is a non-cryptographic djb2 hash for corruption/tamper *detection* only, not prevention (a determined player can still recompute a valid checksum for edited data, since there's no secret key). True cryptographic hashing (e.g. Web Crypto SHA-256) was considered but deferred, since it would require making the save pipeline asynchronous and wouldn't add tamper-prevention anyway without server-side verification.
- Isolated runtime testing of DecaySystem's clamp-at-zero behavior was deferred, since createGame() doesn't currently support custom starting state/entities for testing. The clamp logic was verified by code review and by the production+decay net-rate test. A proper test suite (with the ability to construct custom test states) is planned as a future slice.
- number | BigNumber ergonomics currently only apply to Value/Formula. Component fields like ProductionComponent.rate and DecayComponent.rate are still explicit BigNumber construction in game definitions (buildings.ts) — this remains deferred, lower priority now that Definitions/Value cover the main content-authoring surface.
- **Production/Decay unification.** `ProductionSystem` and `DecaySystem` are structurally near-duplicate (same entity loop, component lookup, target-string convention, modifier fold, resource write — differing only in component type, target prefix, and a sign/clamp step). Leading hypothesis (not yet acted on): collapse into a single `RateComponent`/`RateSystem`, where "production" is a positive rate and "decay" is a negative rate, and `clampAtZero` generalizes into a signed clamp usable to cap either direction. Deliberately deferred — only two data points exist, and the right shared shape isn't reliably visible until a third rate-like system (regeneration, compounding growth, research progress, etc.) either confirms or breaks the hypothesis. If pursued, will also need an ergonomic game-facing layer, since a signed `rate: -5` is less self-documenting in game definitions than the current explicit `production: {...}` / `decay: {...}` component names
Slice 10 raised a related question — whether purchasable/quantitative content (`ProducerDefinition`) should also unify sign-agnostically alongside `ProductionComponent`/`DecayComponent` if this is ever pursued (e.g. purchasable upkeep/decay sources) — deliberately not acted on now, since the trigger condition (a third rate-like system) still hasn't been met; noted here so it's decided as one coherent pass alongside the original hypothesis, not piecemeal.
- **Modifier priority/ordering.** `ModifierSystem.getEffectiveValue()` currently folds modifiers in a fixed three-stage order (add → multiplyAdditive → multiplyCompound) with no way for an individual modifier to demand a different position — e.g. a prestige-style bonus that must apply *last*, after every other modifier, regardless of type. Considered and deliberately deferred during Slice 9 design (no concrete upgrade needs it yet); the well-scoped follow-up is a `priority?: number` field on `ModifierDefinition`, sorted before folding.
- **Bulk purchasing (buy x1/x10/x100/max).** `UpgradeAPI.purchase()` currently only supports single-level purchases. Nearly all idle games require lot-purchasing (buy 10 at once, buy the maximum affordable) — this will need closed-form cost-summation math (summing a geometric-series-shaped cost curve analytically, rather than looping purchase() calls one level at a time, consistent with the engine's "mathematical advancement first" philosophy from design.md). This is a concrete, near-term follow-up slice, not speculative — flagged as something the engine will definitely need. May also reopen the question of whether `GameState.upgrades` levels need to move off plain `number` if a real game's level counts approach `Number.MAX_SAFE_INTEGER` (~9 quadrillion) — deferred for now since single-purchase increments can't reach that range in practice, but bulk purchasing changes the growth rate of level counts and is worth re-examining once implemented.
- **Extending Definitions with game-specific metadata.** Documented as a supported pattern in design.md (typed interface extension, e.g. `interface GameUpgradeDefinition extends UpgradeDefinition {...}`) rather than an engine feature — no engine changes needed unless a concrete need for engine-level involvement emerges.
- **Purchase generalization — partially resolved in Slice 10.** `Purchasable` now exists and is shared by `UpgradeDefinition`, `EntityDefinition`, and `ProducerDefinition`; cost-check/deduct logic is unified via the standalone `evaluatePurchase()`. What's still hardcoded: `Engine.purchase()` only wraps `UpgradeAPI.purchase()` — there's no generic `Engine`-level purchase facade covering `ProducerAPI`/a future entity-acquisition API. Revisit once real producer (or purchasable entity) content actually needs to be purchased through `Engine` rather than an API directly.
- **Engine facade organization.** `Engine` is deliberately the session-level facade — it binds a live GameState so callers don't have to thread `state` through every API call by hand, which is the real justification for its existence alongside (not instead of) the stateless, reusable APIs. Currently a flat list of top-level actions (`tick`, `loadOffline`, `save`, `load`, `purchase`). If this list grows past roughly 6-8 methods, revisit whether namespacing (e.g. `engine.saves.*`) or further purchase-generalization (fewer, broader methods) is the better fix — namespacing alone doesn't reduce duplication, so prefer generalizing content-type-specific methods over nesting them where possible.
- **Extract engine-wiring boilerplate into a `buildStandardEngine()` factory.** `GameSetup.ts` currently mixes two things that don't need to live together: (1) engine plumbing that's identical for every game (constructing ResourceAPI, ComponentAPI, UpgradeAPI, ModifierSystem, ProductionSystem, DecaySystem, Simulation, Clock, Serializer, SaveEncoder, LocalStorageAdapter, SaveManager, Engine, in the right dependency order), and (2) actual game-specific content (which entities/upgrades exist). A game author currently has to read/edit all of (1) just to touch (2). Planned fix: an engine-owned factory function (e.g. `engine/core/EngineBuilder.ts` → `buildStandardEngine(options)`) that takes a small options object (entities, upgrades, saveKey, saveVersion, etc.) and internally does all the category-1 wiring, leaving a game's own `GameSetup.ts` reduced to just defining content and calling the factory. Deliberately not doing this immediately: it's a refactor of already-verified working code (better done as its own clean, isolated, re-verified slice rather than tacked onto Slice 9), and the right shape of "standard options" isn't fully stable yet (still only one example game, and known upcoming changes like bulk purchasing may change what the standard setup needs). No separate "buildCustomEngine" is needed alongside it — manual construction (what GameSetup.ts already does today) *is* the custom path; the factory is purely an added convenience for the common case, not a replacement for direct access to the composable API/system classes. Revisit once the standard-options shape feels stable (e.g. after bulk purchasing and/or a second example game exist to compare against).
